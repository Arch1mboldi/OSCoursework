本次课程设计选取实验三"Linux 系统调用"，要求在 Linux 6.18 内核中新增 3 个自定义系统调用，并编写用户态 ncurses 终端程序实现进程全量监控。阅读和分析的内核模块及相关文件如下：

- `Documentation/RCU/listRCU.rst` — RCU 保护的链表遍历规范

## 二、Linux 内核理解

### 2.1 系统调用流程框架图

```
用户态 (Ring 3)                        内核态 (Ring 0)
──────────────                        ──────────────────

用户程序调用 syscall(SYS_proc_collect, ...)
    │
    ▼
glibc 封装: mov $470, %rax
           mov $buf, %rdi
           mov $count, %rsi
           mov $&ret, %rdx
           syscall              ← CPU 指令
    │                              │
    │   ┌──────────────────────────┘
    │   ▼
    │  CPU 硬件自动完成:
    │    RCX ← RIP  (保存返回地址)
    │    R11 ← RFLAGS
    │    RIP ← MSR_LSTAR  (跳转到内核入口)
    │    CS  ← __KERNEL_CS
    │    SS  ← __KERNEL_DS
    │    CPL ← 0  (切换到 Ring 0)
    │
    ▼                              │
  (返回用户态)                     ▼
    ▲                         entry_SYSCALL_64 (汇编)
    │                              │
    │                         swapgs (切换到内核 GS)
    │                         保存用户栈指针到 per-CPU
    │                         切换到内核栈
    │                         构造 pt_regs 帧 (保存全部寄存器)
    │                              │
    │                              ▼
    │                         do_syscall_64(regs)
    │                              │
    │                         sys_call_table[regs->ax]  ← 查表
    │                         ─────────────────────────
    │                         regs->ax = 470
    │                         → sys_proc_collect(regs->di, regs->si, regs->dx)
    │                              │
    │                              ▼
    │                         kernel/proc_monitor.c
    │                         ┌─────────────────────┐
    │                         │ read_lock(tasklist) │
    │                         │ for_each_process()  │
    │                         │ copy_to_user()      │
    │                         │ read_unlock(tasklist)│
    │                         └─────────────────────┘
    │                              │
    │                              ▼
    │                         sysretq  ← CPU 指令
    │                         ┌──────────────────────┐
    │                         │ RIP ← RCX            │
    │                         │ RFLAGS ← R11         │
    │                         │ CPL ← 3 (回到 Ring 3)│
    │                         └──────────────────────┘
    │                              │
    └──────────────────────────────┘
```

**关键要点**：
1. x86_64 上 syscall/sysret 指令对替代了传统的 int 0x80，速度更快（无中断门查表开销）
2. 系统调用号通过 rax 传递，参数通过 rdi/rsi/rdx/r10/r8/r9 传递（遵循 x86_64 ABI）
3. 内核态不能直接解引用用户态指针——必须通过 copy_to_user() / copy_from_user()（§2.3 详细解释）
4. MSR_LSTAR（Long System Target Address Register）是 x86_64 特有的 MSR 寄存器，在系统启动时由 syscall_init() 设置为 entry_SYSCALL_64 地址

### 2.2 核心数据结构的解析

#### 2.2.1 task_struct — 进程描述符

task_struct 是 Linux 内核中最重要的数据结构之一，每个进程/线程都对应一个 task_struct 实例。它定义在 `include/linux/sched.h`，包含数百个字段。本项目主要使用以下字段：

**进程标识**：
- `pid_t pid` — 进程 ID（实际上在 6.x 内核中 task_struct 使用 struct pid* 间接引用，通过 task_tgid_nr() 宏获取数字 PID）
- `struct task_struct *real_parent` — 真实父进程指针（区别于 ptrace 时的临时父进程 parent）

**进程状态（Linux 5.14+ 拆分）**：
- `unsigned int __state` — 调度状态：TASK_RUNNING(0x00)、TASK_INTERRUPTIBLE(0x01)、TASK_UNINTERRUPTIBLE(0x02)、__TASK_STOPPED(0x04)、__TASK_TRACED(0x08)
- `int exit_state` — 退出状态：EXIT_DEAD(0x10)、EXIT_ZOMBIE(0x20)
- `__state` 可带标志位：TASK_WAKEKILL(0x0080)、TASK_NOLOAD(0x0400)、TASK_WAKING(0x0100)

**内存描述**：
- `struct mm_struct *mm` — 用户态地址空间描述符。内核线程此字段为 NULL（无用户态地址空间）
- `mm->total_vm` — 虚拟内存页数，乘以 PAGE_SIZE(4KB) 得字节数
- 常驻内存集通过 `get_mm_rss(mm)` 获取（RSS 页数）

**CPU 时间**：
- CPU 时间以纳秒精度存储在 `task->thread_info.utime/stime` 或 `se.sum_exec_runtime` 中
- 通过 `task_cputime(task, &ut_ns, &st_ns)` 读取（在 VIRT_CPU_ACCOUNTING_GEN 配置下）

**信号与线程**：
- `struct signal_struct *signal` — 线程组共享的信号描述符。需判空（刚创建的进程可能尚未分配）
- `signal->nr_threads` — 线程组内线程计数。若 signal 为空则视为单线程

#### 2.2.2 进程链表结构

Linux 内核通过双向循环链表管理所有进程。链表头是 `init_task`（PID 0 的 task_struct），所有其他进程通过 `tasks` 成员（`struct list_head`）链接：

```
init_task ←→ task1 ←→ task2 ←→ ... ←→ taskN ←→ init_task
   (PID 0)     (PID 1)   (PID 2)            (PID N)
```

遍历宏：
- `for_each_process(task)` — 展开为 `for (task = &init_task; (task = next_task(task)) != &init_task; )`
- `next_task(task)` — 通过 `list_next_entry(task, tasks)` 获取下一节点，本质是 `container_of(task->tasks.next)`

链表受 RCU（Read-Copy-Update）和 tasklist_lock 双重保护：
- RCU 保证读者在无锁遍历时不会看到正在被释放的节点（宽限期机制延迟回收）
- tasklist_lock 是读写自旋锁：读者用 read_lock()，写者（fork/exit）用 write_lock()

#### 2.2.3 自定义数据结构 — ABI 约定

3 个自定义结构体作为内核与用户态的数据交换格式，定义在两边头文件中保持字节级一致：

**proc_info**（x86_64 下 sizeof=80）：
```
偏移   大小  字段         说明
0      4     pid          int，线程组 ID
4      4     ppid         int，父进程 ID
8      16    comm[16]     char 数组，进程名（最多 15 字符 + NUL）
24     4     state        int，__state | exit_state 合并值
28     4     (padding)    对齐 unsigned long (8B)
32     8     utime        unsigned long，用户态 CPU 滴答
40     8     stime        unsigned long，内核态 CPU 滴答
48     8     vsize        unsigned long，虚拟内存字节
56     8     rss          unsigned long，常驻内存页数
64     4     nice         int，-20 到 19
68     4     num_threads  int，线程组线程数
72     4     uid          uid_t，真实用户 ID
76     4     (padding)    对齐到 8 字节边界
```

**proc_tree_node**（x86_64 下 sizeof=28）：pid(4B)+ppid(4B)+comm(16B)+level(4B)

**proc_stat**（x86_64 下 sizeof=36）：9 个 int 计数器，记录各状态/类型的进程数

### 2.3 核心函数流程说明

#### 2.3.1 sys_proc_collect (470) — 进程信息采集

```
sys_proc_collect(user_buf, max_count, &ret_count)
│
├─ 参数校验: user_buf==NULL || max_count<=0 → 返回 -EINVAL
│
├─ read_lock(&tasklist_lock)       ← 获取读锁
├─ task = next_task(&init_task)    ← 跳过 init_task (PID 0)
│
├─ while (task != &init_task):
│   │
│   ├─ memset(&kinfo, 0, 80)      ← 清空临时 proc_info
│   ├─ 采集进程名: get_task_comm(kinfo.comm, task)
│   ├─ 采集 PID/PPID: task_tgid_nr(task), task_tgid_nr(task->real_parent)
│   ├─ 采集状态: kinfo.state = task->__state | task->exit_state
│   ├─ 采集 CPU 时间: task_cputime() → nsec → nsec_to_clock_t() → USER_HZ ticks
│   ├─ 采集内存: task->mm->total_vm << PAGE_SHIFT → vsize
│   │             get_mm_rss(task->mm) → rss
│   │             (内核线程 mm==NULL → 填 0)
│   ├─ 采集其他: task_nice(), signal->nr_threads, from_kuid_munged(...)
│   │
│   ├─ next = next_task(task)      ← ★ 持锁保存后继指针（关键！）
│   ├─ read_unlock(&tasklist_lock) ← 释放锁
│   │
│   ├─ copy_to_user(&buf[i], &kinfo, 80)  ← 无锁拷贝（可能睡眠）
│   ├─ count++
│   │
│   ├─ read_lock(&tasklist_lock)   ← 重新加锁
│   └─ task = next                 ← 使用保存的指针继续遍历
│
├─ read_unlock(&tasklist_lock)
└─ copy_to_user(ret_count, &count, 4)  → 返回 0
```

**关键设计点**：
1. 遍历中使用 `next_task(task)` 保存后继而非 `task->tasks.next`：释放锁后 task 可能被 RCU 回收，`task->tasks.next` 是 UAF（Use-After-Free）。保存的 `next` 是独立指针值，目标进程若未被释放则持续有效
2. copy_to_user() 在锁外执行：避免在 RCU 读临界区睡眠（§5.3 详述 RCU 约束）
3. 逐个拷贝：避免在内核栈上分配 `struct proc_info[8192] = 640KB`，远超 THREAD_SIZE(16KB)
4. 错误恢复：copy_to_user() 失败时尽力返回已成功拷贝的条目数

#### 2.3.2 sys_proc_snapshot (471) — 进程树快照

流程与 sys_proc_collect 相同（lock-drop-copy-relock），区别在于：
- 填充 proc_tree_node 而非 proc_info（字段更少：PID, PPID, comm, level）
- 调用 `compute_level(task)` 计算树深度：从 task 沿 real_parent 向上走到 init_task，累计步数
- compute_level 在持锁期间调用（上溯过程中遍历的是活跃 task_struct，安全）

#### 2.3.3 sys_proc_stat (472) — 进程统计

```
sys_proc_stat(struct proc_stat *stat)
│
├─ read_lock(&tasklist_lock)       ← 全程持锁
│
├─ for_each_process(task):
│   │
│   ├─ kstat.total_processes++
│   │
│   ├─ 状态分类 (按优先级):
│   │   ├─ task->exit_state & EXIT_ZOMBIE  → kstat.zombie_processes++
│   │   ├─ task->exit_state & EXIT_DEAD    → skip (不计数)
│   │   └─ 否则:
│   │       ├─ task->__state & TASK_NOLOAD → kstat.idle_processes++
│   │       └─ 否则:
│   │           s = task->__state & ~(TASK_WAKEKILL | TASK_WAKING)
│   │           switch (s):
│   │             TASK_RUNNING        → kstat.running_processes++
│   │             TASK_INTERRUPTIBLE  → kstat.sleeping_processes++
│   │             TASK_UNINTERRUPTIBLE→ kstat.uninterruptible++
│   │             __TASK_STOPPED      → kstat.stopped_processes++
│   │             __TASK_TRACED       → kstat.stopped_processes++
│   │             default             → kstat.idle_processes++
│   │
│   └─ 线程类型:
│       ├─ task->mm == NULL → kstat.kernel_threads++
│       └─ task->mm != NULL → kstat.user_threads++
│
├─ read_unlock(&tasklist_lock)
└─ copy_to_user(stat, &kstat, 36)   ← 循环后拷贝，安全
```

**关键设计点**：
1. stat 是唯一全程持锁的系统调用：循环中只做整数运算，无 copy_to_user()，不违反 RCU 约束
2. 状态分类必须用位掩码（不是精确匹配 switch）：`__state` 可带 TASK_WAKEKILL(0x0080) 等标志位。若 `switch(task->__state)` 精确匹配，0x0081 (INTERRUPTIBLE|WAKEKILL) ≠ 0x0001，落入 default → 误判为 idle
3. TASK_NOLOAD 单独检测：TASK_IDLE 语义上 = UNINTERRUPTIBLE | NOLOAD，应归 idle
4. EXIT_DEAD 跳过不计入任何活跃分类：进程正在被内核回收，与 EXIT_ZOMBIE（等待父进程 wait()）不同

## 三、系统设计方案

### 3.1 数据结构设计

采用软件工程中 ABI（Application Binary Interface）约定的设计思想，内核侧与用户态侧各自维护完全一致的数据结构定义，确保跨特权级的数据交换安全。

**设计原则**：
1. 值类型传递：所有字段均为值类型（int, unsigned long, char[]），不包含指针，杜绝内核地址泄露
2. 字段精选：从 task_struct 数百字段中选取 11 个维度，覆盖进程监控核心需求——身份（PID/PPID/UID）、状态、时间（utime/stime）、内存（vsize/rss）、调度（nice）、并发度（num_threads）
3. LP64 对齐：x86_64 ABI 下 int 为 4 字节，unsigned long 为 8 字节，struct proc_info 的 padding 由编译器自动填充（偏移 28 和 76），总大小 80 字节
4. 版本兼容：字段顺序固定，不依赖内核结构体布局细节，内核升级后只需重新编译无需修改代码

**数据结构汇编**：
- struct proc_info：单个进程完整信息，80 字节。用于 sys_proc_collect 的数组元素
- struct proc_tree_node：进程树节点，28 字节。pid/ppid 重建父子关系，level 支持缩进渲染
- struct proc_stat：聚合统计，36 字节。9 个 int 计数器，一次系统调用完成全部统计

详细的内存布局和数据字段对照表见 supplementary.md。

### 3.2 系统功能设计

系统划分为两大模块：

**内核模块（kernel/proc_monitor.c）**：

| 系统调用          | 编号  | 参数                             | 返回值               | 功能                |
| ------------- | --- | ------------------------------ | ----------------- | ----------------- |
| proc_collect  | 470 | user_buf, max_count, ret_count | 0/-EINVAL/-EFAULT | 采集所有进程的 proc_info |
| proc_snapshot | 471 | user_buf, max_count, ret_count | 0/-EINVAL/-EFAULT | 返回进程树拓扑           |
| proc_stat     | 472 | stat                           | 0/-EINVAL/-EFAULT | 返回分类统计            |

**用户态监控程序（user_app/proc_monitor.c）**：

- 数据采集：通过 syscall(SYS_proc_xxx) 调用自定义系统调用，记录每次调用的延迟（微秒级）
- 4 面板 ncurses TUI：主视图区、详情行、系统调用诊断行、按键提示行
- 5 种视图模式：LIST（紧凑列表，状态颜色高亮）、DEBUG-TABLE（所有原始数值）、TREE（UTF-8 树线 + 按深度着色）、HEX-DUMP（选中进程 80 字节原始 dump + 字段偏移标注）、SYSCALL（系统调用诊断面板 + 交叉验证）
- 排序：PID / CPU% / 内存 / 进程名，支持升降序
- 过滤：名称子串（大小写不敏感）、状态字符（=R/S/D/T/Z）、精确状态十六进制（=0x22）、精确 PID（:1234）
- 导出：一键导出全部三种格式——CSV（结构化进程列表）、DOT（Graphviz 进程树图）、BIN（二进制内存 dump，含 magic 头）
- 交叉验证：在 SYSCALL 面板底部对比 proc_info[] 自统计与 proc_stat 返回值，检测状态分类的正确性
- 系统调用验证程序（test_syscall.c）：独立于主程序的测试工具，用 64 容量 buffer 验证 3 个系统调用的注册正确性

### 3.3 系统流程设计

**总体运行时流程**：

```
启动 → 初始化 ncurses (4 面板)
     → 首次数据采集 (update_data)
       ├─ fetch_procs()  → syscall(470)  记录耗时
       ├─ fetch_tree()   → syscall(471)  记录耗时
       └─ fetch_stat()   → syscall(472)  记录耗时
     → 排序 (sort_procs)
     → 保存帧 (save_prev_cpu: 记录当前 CPU 滴答值)

主循环 (每秒):
  渲染 → wgetch(1 秒超时)
    ├─ 超时 (1000ms): auto-refresh
    │   save_prev_cpu() → update_data() → sort_procs() → 重绘
    │
    ├─ 按键 v/V:    切换视图模式
    ├─ 按键 1-5:    直达指定视图
    ├─ 按键 ↑↓/jk: 移动光标
    ├─ 按键 n/p:    HEX-DUMP 视图切换进程
    ├─ 按键 Tab:    切换排序字段
    ├─ 按键 s:      切换升降序
    ├─ 按键 /:      进入过滤输入模式
    ├─ 按键 r:      清除过滤器
    ├─ 按键 x:      导出全部格式 (CSV+DOT+BIN)
    ├─ 按键 h/?:    显示帮助覆盖层
    └─ 按键 q/ESC:  退出
```

**数据流**：

```
内核 task_struct 链表
  → [read_lock + 遍历 + 提取字段]
  → proc_info[] / proc_tree_node[] / proc_stat (内核栈局部变量)
  → copy_to_user()
  → 用户态 g_procs[] / g_tree_nodes[] / g_stat (全局数组)
  → sort + filter
  → 5 视图渲染器 (ncurses wprintw)
  → 终端输出
```

**CPU% 计算数据流**：

```
第 N 帧: g_prev[].total = utime + stime (当前累计滴答)
         g_tv_prev  ← 采集时间戳

第 N+1 帧: 采集新数据
         g_tv_curr  ← 新采集时间戳
         elapsed = g_tv_curr - g_tv_prev (秒)

         cpu_pct(pid, cur_total):
           prev_total = bsearch(pid, g_prev[])  // 查找上一帧滴答
           delta = cur_total - prev_total
           return delta / (clk_tck × elapsed) × 100%
```

### 3.4 算法设计

**3.4.1 进程遍历与锁算法 (lock-drop-copy-relock)**

```
输入: 用户态 buffer 指针, 最大条目数
输出: 填充的 proc_info 数组

算法:
1. read_lock(&tasklist_lock)
2. task ← next_task(&init_task)
3. while task ≠ &init_task:
4.    临时变量 kinfo ← 从 task 提取的 proc_info
5.    next ← next_task(task)          // 持锁保存后继
6.    read_unlock(&tasklist_lock)     // 释放
7.    copy_to_user(&buf[i], &kinfo)   // 可睡眠
8.    count++
9.    read_lock(&tasklist_lock)       // 重加锁
10.   task ← next                     // 使用保存的后继
11. read_unlock(&tasklist_lock)
12. 返回 count
```

时间复杂度：O(N)，其中 N 为进程总数。N 次 read_lock/unlock 对（每次 O(1)），N 次 copy_to_user（瓶颈在内存拷贝，80 字节每次）。

空间复杂度：内核栈使用 O(1)（仅一个 struct proc_info 临时变量，80 字节）。用户态 buffer 由调用者分配。

**3.4.2 进程树深度计算算法**

```
compute_level(task):
  level ← 0
  parent ← task->real_parent
  while parent ≠ NULL and parent ≠ &init_task:
    level++
    parent ← parent->real_parent
  return level
```

时间复杂度：O(H)，其中 H 为进程树高度（通常 ≤ 10）。调用者持有 tasklist_lock 读锁保证上溯过程中 task_struct 不会被释放。

**3.4.3 进程状态分类算法（位掩码方法）**

```
输入: task->__state (unsigned int), task->exit_state (int)
输出: 状态分类类别

算法:
1. if exit_state & EXIT_ZOMBIE (0x20):
      return ZOMBIE
2. if exit_state & EXIT_DEAD (0x10):
      skip (不计数)
3. // 活跃进程分类
4. if __state & TASK_NOLOAD (0x0400):
      return IDLE              // TASK_IDLE = UNINTERRUPTIBLE | NOLOAD
5. s ← __state & ~(TASK_WAKEKILL | TASK_WAKING)  // 清除标志位
6. switch (s & 0x03):         // 基础状态是低 2 位
      case 0: return RUNNING
      case 1: return SLEEPING  (INTERRUPTIBLE)
      case 2: return DISK_WAIT (UNINTERRUPTIBLE)
7. if s & __TASK_STOPPED (0x04) or s & __TASK_TRACED (0x08):
      return STOPPED
8. return IDLE                 // TASK_NEW (0x0800) 等过渡状态
```

正确性保证：步骤 5 的掩码操作确保 0x0081 → 0x0001，正确识别为 SLEEPING 而非 default→IDLE。

**3.4.4 CPU% 差值计算算法**

```
输入: pid (目标进程), cur_total (当前累计 CPU 滴答)
输出: CPU% (0-100×CPU核数)

数据结构: g_prev[] 数组 (按 PID 排序), g_tv_prev/g_tv_curr (时间戳)

1. if g_has_prev == 0:  return 0.0    // 第一帧: 无历史数据
2. prev_total ← bsearch(pid, g_prev[])  // 二分查找上一帧滴答
3. if not found or prev_total > cur_total:  return 0.0  // PID 复用
4. delta ← cur_total - prev_total
5. elapsed ← g_tv_curr - g_tv_prev (秒, 微秒精度)
6. if elapsed ≤ 0: return 0.0
7. return delta / (clk_tck × elapsed) × 100.0
```

时间复杂度：每帧排序 O(N log N)，每次查询 O(log N)（二分查找）。空间复杂度 O(N)（g_prev[] 与 g_procs[] 等大）。

**3.4.5 排序与过滤算法**

排序：使用标准库 qsort()，时间复杂度 O(N log N)。比较函数根据 g_sort_field 分发：
- PID：直接比较 pid 字段
- CPU%：调用 cpu_pct() 获取增量百分比再比较
- 内存：比较 rss 字段
- 进程名：使用 strcmp() 比较 comm 字符串

过滤：线性扫描 O(N)，每条记录调用 match_filter()。支持的子模式：
- 普通字符串：子串匹配（大小写不敏感，线性扫描 comm 字段）
- `=R`：状态字符精确匹配（O(1)）
- `=0xNN`：state 十六进制值精确匹配（O(1)，调试用）
- `:1234`：PID 精确匹配（O(1)）

## 五、小结

### 5.1 课程设计中遇到的问题及解决办法

**问题 1：进程状态分类系统性错误**

交叉验证面板显示大量 MISMATCH（如 sleeping: proc_info 统计 224 但 proc_stat 报告 211，idle: 统计 0 但报告 167）。经深入排查，根因在于 Linux 5.14+ 的 `task->__state` 采用了位掩码编码——`TASK_INTERRUPTIBLE = 0x0001`，但实际运行中的进程 `__state` 可能为 `0x0081`（INTERRUPTIBLE | TASK_WAKEKILL）。内核侧的 `switch(task->__state)` 精确匹配无法识别带标志位的值，导致大量进程被误判为 idle。

**解决办法**：将内核和用户态两侧的分类逻辑全部改为位掩码方式。先检查 `TASK_NOLOAD(0x0400)` 将空闲内核线程归入 idle，再用 `__state & ~(TASK_WAKEKILL | TASK_WAKING)` 清除标志位，最后 `switch(s & 0x03)` 分类。用户态交叉验证同步修改，增加 NOLOAD 检测和 WAKEKILL 掩码。修复后所有类别交叉验证全部 OK。

**问题 2：CPU% 计算异常**

第一帧刷新后 CPU% 显示高达数百甚至数千百分比。排查发现时间戳记录位置错误——`gettimeofday()` 在渲染函数 `save_prev_cpu()` 中调用（渲染前），而非在数据采集函数 `update_data()` 中调用。由于渲染和数据采集几乎同时发生，`elapsed ≈ 0`，导致 `delta_ticks / (clk_tck × 0.00001)` 产生巨大的百分比。

**解决办法**：将时间戳记录从 `save_prev_cpu()` 移至 `update_data()` 内部，在 `fetch_procs()`/`fetch_tree()`/`fetch_stat()` 调用之前记录 `g_tv_prev` ← `g_tv_curr`，然后记录新的 `g_tv_curr`。确保 `elapsed` 反映的是两次数据采集之间的真实时间差。

**问题 3：排序度量与显示列不一致**

按 CPU% 列排序时，实际排序顺序与终端显示的 CPU% 值不一致。根因在于排序比较函数 `cmp_cpu` 使用 `utime + stime`（进程启动以来的累计 CPU 滴答）作为排序键，而显示列调用 `cpu_pct()` 展示的是增量百分比（基于前后两帧差值）。对于运行时间长的守护进程，累计滴答很大但增量很小，导致排序结果与视觉预期不符。

**解决办法**：将 `cmp_cpu` 改为调用 `cpu_pct()`，使排序键与显示值完全一致。

**问题 4：HEX-DUMP 视图导航设计缺陷**

HEX-DUMP 视图中，上下方向键的语义存在冲突：既需要滚动 hex 行（查看 80 字节的不同片段），又需要切换进程（查看不同进程的 dump）。原始设计使用同一个 `g_selected` 变量同时表示进程索引和 hex 行号，导致上下键切换了进程而非滚动 hex 行。

**解决办法**：引入独立的 `g_hex_selected` 变量跟踪被 dump 的进程索引，将 `g_selected` 在 HEX-DUMP 视图中重新定义为 hex 行光标。新增 `n` / `p` 键专门用于切换被 dump 的进程。进入/离开 HEX-DUMP 视图时做 `g_selected` ↔ `g_hex_selected` 的同步。


### 5.2 课程设计还存在的问题

1. **系统调用号未标准化**：当前使用 470/471/472 作为系统调用号，这些编号在 Linux 主线内核中未被分配，但在生产环境中可能与未来主线分配冲突。正式部署应使用动态分配机制或选择预留范围。

2. **最大进程数硬编码**：`MAX_PROCS = 8192` 是静态上限，对于超过 8192 个进程的高负载服务器场景不够用。改进方向是支持动态数组扩展，或分批采集。

3. **缺少权限控制**：目前所有用户均可调用 3 个自定义系统调用读取全部进程信息。虽然提取的字段不包含敏感数据，但在安全敏感环境中应增加 CAP_SYS_PTRACE 权限检查。

4. **缺少守护进程模式**：用户态程序仅支持交互式 TUI 模式，缺乏后台守护进程 + 日志记录模式，不便于长时间监控和事后分析。

5. **进程树快照与进程信息采集的时间不一致**：`fetch_procs()` 和 `fetch_tree()` 是两次独立的系统调用，之间存在微小的时间窗口。若在此窗口内进程创建或退出，proc_info 和 proc_tree_node 的数据可能不完全对应。

6. **跨平台兼容性**：当前仅在 x86_64 架构上验证。ARM64/riscv64 等其他架构的 LP64 ABI 对齐规则可能不同，结构体布局需重新验证。

这次课程设计也让我深刻体会到编程中细节的重要性。一个看似简单的 `switch(task->__state)` 精确匹配，因为忽略了 `TASK_WAKEKILL` 等标志位，导致数百个进程的分类错误。这个问题直到实现了交叉验证面板才被发现——如果没有系统的自检机制，这个错误可能会一直隐藏，让统计结果在"看起来正常"的情况下悄悄地出错。这让我意识到：任何一个与数据编码相关的假设都必须显式验证，标志位是位掩码而不是枚举值这样的领域知识必须反映在代码实现中。

在用户态程序开发方面，ncurses 编程是全新的体验。在图形界面盛行的今天，用纯文本终端构建交互式监控工具是一种非常"硬核"的程序设计方式。4 面板布局的空间管理、UTF-8 树线字符的终端兼容性、HEX-DUMP 视图的字节级可视化、过滤模式下的逐字符输入处理——这些细节让一个"能用"的程序变成一个"好用"的工具。特别是将调试信息（系统调用耗时、结构体大小、交叉验证）直接集成到 TUI 中，而不是依赖外部日志文件，这种设计思路让开发迭代效率大大提高。
